# ТЗ семестрового проекта по разработке языка программирования и его компилятора v0.1

# Идея для проекта: 
"Небольшой строго типизированный язык программирования с выводом типов для переменных, пользовательскими структурами и базовой генерацией кода."

# Описание проекта: 
Целью проекта является разработка небольшого компилируемого языка программирования и реализация компилятора для него на языке C++.

Разрабатываемый язык имеет C-подобный синтаксис и поддерживает базовые конструкции процедурного программирования: переменные, арифметические и логические выражения, функции, условные операторы, циклы, пользовательские структуры и вывод значений. Отличительной особенностью языка является строгая статическая типизация и ограниченный вывод типов для локальных переменных.

Компилятор должен реализовывать полный базовый pipeline обработки программы:
исходный текст → лексический анализ → синтаксический анализ → семантический анализ → генерация кода

Ключевое решение это генерировать ограничения по типам внутри функции и решать их унификацией (без обобщения/полиморфизма), что практически сводится к «Algorithm W без generalization».  

Для генерации кода рассматриваются три бэкенда: генерация C++ исходников, генерация LLVM IR и генерация байткода для собственной VM. Рекомендация по умолчанию при приоритете «простота реализации и понятность»: генерация C++ (как транспиляция) — минимальный объём инфраструктуры, хорошая дебажимость и быстрый путь к рабочему прототипу.

# Цели:
Проект должен продемонстрировать:
    проектирование собственного языка программирования;
    построение формальной грамматики;
    реализацию лексического и синтаксического анализатора;
    реализацию семантической проверки и проверки типов;
    реализацию ограниченного вывода типов;
    генерацию исполнимого кода через транспиляцию в C++.

# Характеристика:
Язык является:
    статически типизированным;
    строго типизированным;
    компилируемым;
    императивным по стилю;
    C-подобным по синтаксису.

В языке поддерживаются:
    встроенные типы int, float, bool, void;
    литералы;
    арифметические, логические и сравнительные выражения;
    переменные let и var;
    функции;
    if, while, return;
    встроенная операция print;
    пользовательские структуры struct;
    доступ к полям через оператор ..

# Ограничения:
Для упрощения реализации в базовой версии языка:
    отсутствуют обобщённые типы (generics, templates);
    отсутствует параметрический полиморфизм;
    отсутствует перегрузка функций;
    отсутствуют указатели и ссылки как типы;
    отсутствуют пространства имён и модули;
    отсутствуют массивы;
    методы структур являются синтаксическим сахаром над функциями с параметром self.

# Требования, которые я включу в проект:
Язык предназначен для реализации компилятора «полного цикла», то есть лексинг → парсинг → семантика/типы → IR → простые оптимизации → кодоген.

    Функциональные требования (минимальное ядро):
        Числовые типы: int, float (строго различаются; неявных преобразований нет).
        Базовые выражения: литералы, арифметика, сравнения, логика (минимум для условий), доступ к полям через ., вызовы функций/методов.
        Переменные: объявление, присваивание; type inference для локальных переменных при наличии инициализатора.
        Функции: параметры с явно указанными типами; возвращаемый тип явно указан.
        Управляющие конструкции: if, while, return.

    Нефункциональные требования:
        Строгая статическая типизация: ошибка типов — ошибка компиляции.
        Детерминированность вывода типов: при неоднозначности — диагностируемая ошибка, а не «магия».
        Простая реализация: ручной парсер (рекурсивный спуск + Pratt для выражений), явный AST и явная фаза типизации. 

    Осознанные ограничения (для упрощения):
        Нет обобщённых типов (generics/templates) и параметрического полиморфизма.
        Нет перегрузки пользовательских функций.
        Нет указателей/ссылок как типов (чтобы не вводить алиасинг и правила владения).
        Методы структур — синтаксический сахар над функциями с первым параметром self (подробно в семантике).
    
# Синтаксис и лексика
Ключевые слова: struct, fn, let, var, if, else, while, return, true, false, print.
Грамматика обязана покрывать:
- структуру программы и точку входа (`main`)
- все объявления — переменных, функций, структур, синонимов типов, пространств имён
- все выражения — арифметические, логические, сравнения, индексирование, доступ к полям
- все конструкции управления потоком — блоки, `if/else`, `while`, `break`, `continue`, `return`, нулевая инструкция (инструкция без эффекта)
- литералы массивов и структур
Литералы:
- IntLit: последовательность цифр (0|[1-9][0-9]*).
- FloatLit: цифры + . + цифры (минимум одна цифра по обе стороны, чтобы избегать коллизий лексера).
Разделители/операторы: { } ( ) , ; : -> . = + - * / % == != < <= > >= && || !
Таблица приоритетов операторов: такая же как в баше

# Семантика 
Язык обязан определять и будет на моем языке:
- **Стратегия вычисления** — call-by-value: аргументы передаются в функцию по значению
- **Семантику присваивания** — value semantics (копирование по значению)
- **Инициализация переменных** — использование неинициализированной переменной
  является ошибкой компиляции
- **Время жизни переменных** — переменная существует в пределах блока, в котором объявлена
- **Побочные эффекты** — функции могут иметь побочные эффекты (ввод/вывод, изменение переменных)
- **Поведение при ошибках времени выполнения** — при возникновении ошибки
  программа завершается с сообщением вида:
     runtime error: <сообщение> at line <N>
- Обязательно обрабатываются: деление на ноль, выход за границы массива.

- Стратегия вычисления выражений: слева направо, без «ленивости». (Это решение спецификации; важно для предсказуемости побочных эффектов print.)
- Области видимости: лексические, блочные. Имя видимо после объявления; теневание (shadowing) допускается, но повторное объявление в одной области — ошибка.
- Присваивание: if/while: условие должно иметь тип bool. Никаких неявных преобразований int→bool нет.
- return:
        В функции с типом результата void допускается return;.
        В функции с типом результата T != void обязателен return expr; по всем путям исполнения (минимальная проверка достижимости — в семантической фазе).
- print(expr);: допускается для int/float/bool. Печать структур в базовой версии запрещена (ошибка типов), чтобы не вводить форматирование как часть ядра языка.

- Ошибки времени выполнения
Целевая платформа не задана, поэтому runtime-модель должна быть минимальной и переносимой. В базовой версии рекомендуется определить:
    деление на ноль → runtime error;
(опционально) переполнение int → либо runtime error, либо «поведение не определено спецификацией» (см. обсуждение в кодогенерации: для C++-бэкенда «определённое поведение при overflow» требует дополнительных мер).

# База
Компилятор должен быть реализован в виде последовательного pipeline из независимых фаз.
Каждая фаза принимает на вход результат предыдущей и не зависит от деталей её реализации.

Исходный код (.lang)
       ↓
   [Лексер]          → поток токенов
       ↓
   [Парсер]          → AST
       ↓
  [Семантика]        → аннотированный AST
       ↓
   [Кодоген]          → x86-64 asm / LLVM IR
       ↓
  Исполняемый файл

обязательно Каждая фаза должна быть реализована отдельным модулем/классом.

# Вывод типов:
- Язык поддерживает ограниченный вывод типов только для локальных переменных внутри тела функции. 
Область применения: Вывод типов разрешён только для объявлений вида:
let x = expr;
var y = expr;
Если тип не указан явно, он определяется по инициализатору.
- Ограничения вывода типов: Вывод типов не применяется к:
параметрам функций;
возвращаемому типу функции;
полям структур.
Следовательно, эти типы должны быть заданы явно.
- Принцип работы:
Вывод типов выполняется в семантической фазе:
по AST строятся ограничения на типы;
ограничения решаются с помощью унификации;
найденный тип подставляется в объявление переменной.
В базовой версии используется мономорфный локальный вывод типов без обобщения.

# Структуры и методы
Поля структуры: Доступ к полям осуществляется через оператор . : p.x. Если поле отсутствует в определении структуры, это является ошибкой компиляции.
Литералы структур: Структура может быть создана литералом вида: Point{ x: 1, y: 2 }. Тип литерала определяется именем структуры.
Методы: Методы структур в базовой версии языка рассматриваются как синтаксический сахар над обычными функциями с первым параметром self.

# Ошибки компиляции и диагностика
Компилятор обязан обнаруживать и диагностировать ошибки на всех этапах обработки программы.
Ошибки делятся на следующие категории:

1. Лексические ошибки
Возникают на этапе лексического анализа, если входной текст не может быть разбит на корректные токены.
    Примеры:
        неизвестный символ;
        некорректный числовой литерал;
        незавершённый комментарий или строковый литерал (если строки будут добавлены позже).
    При обнаружении лексической ошибки компилятор обязан:
        указать позицию ошибки в исходном файле;
        вывести понятное диагностическое сообщение;
        прекратить дальнейшую компиляцию.

2. Синтаксические ошибки
Возникают на этапе парсинга, если последовательность токенов не соответствует грамматике языка.
    Примеры:
        пропущенная ;;
        пропущенная };
        неожиданное ключевое слово или оператор;
        некорректная структура выражения или инструкции.
    При обнаружении синтаксической ошибки компилятор обязан:
        указать точную позицию ошибки;
        по возможности сообщить, какой токен ожидался;
        остановить синтаксический анализ после первой ошибки.

3. Семантические и типовые ошибки
Возникают после успешного построения AST, на этапе семантической проверки и проверки типов.
    Примеры:
        использование необъявленной переменной;
        повторное объявление имени в одной области видимости;
        присваивание значения несовместимого типа;
        применение арифметической операции к bool или struct;
        использование выражения не типа bool в условии if или while;
        неверное количество аргументов функции;
        обращение к несуществующему полю структуры;
        невозможность вывести тип локальной переменной;
        отсутствие return по всем путям в функции с невойдовым типом результата.
    При обнаружении семантической или типовой ошибки компилятор обязан:
        указать место ошибки;
        по возможности показать ожидаемый и фактический тип;
        прекратить компиляцию.

# Генерация кода
В базовой версии проекта генерация кода осуществляется через транспиляцию в C++.
Генератор обязан:
    корректно переводить все конструкции языка;
    сохранять семантику строгой типизации;
    генерировать компилируемый C++ код;
    использовать внешний C++-компилятор для получения исполняемого файла.

# Заключение
Предлагаемый язык представляет собой небольшой, но полноценный учебный компилируемый язык со строгой типизацией, пользовательскими структурами и ограниченным выводом типов. Он достаточно прост для реализации в рамках семестрового проекта и одновременно покрывает ключевые этапы построения компилятора: от лексического и синтаксического анализа до типизации и генерации кода.
