# semantics.md — семантика языка **Astra**

## 1. Назначение документа

Документ определяет смысл всех конструкций языка **Astra** и обязательные проверки,
которые должен выполнять семантический анализатор.

Спецификация рассчитана на то, чтобы компилятор можно было реализовать,
не опираясь на исходный код автора проекта.

---

## 2. Общая семантическая модель

### 2.1. Характер языка

**Astra** — статически типизированный, строго типизированный компилируемый язык
с **value semantics**.

Основные свойства:

- все типы известны на этапе компиляции;
- неявные приведения отсутствуют;
- значения передаются и присваиваются **по значению**;
- аргументы функций передаются **по значению** (**call-by-value**);
- выражения имеют детерминированный порядок вычисления слева направо;
- локальные переменные имеют блочную область видимости.

### 2.2. Модули

Каждый исходный файл является отдельным модулем.
В базовой версии языка все верхнеуровневые объявления модуля считаются видимыми
внутри этого модуля. Межфайловый импорт в базовую версию не входит.

Если компилятор принимает несколько файлов, они объединяются в одну программу,
а полные имена объявлений учитывают пространства имён.

---

## 3. Пространства имён и разрешение имён

### 3.1. Лексическая область видимости

Язык использует **lexical scoping**.

Идентификатор разрешается в ближайшей охватывающей области видимости по следующим правилам:

1. локальная область текущего блока;
2. параметры текущей функции;
3. внешние блоки функции;
4. объявления в текущем пространстве имён;
5. объявления во внешних пространствах имён;
6. встроенные функции.

### 3.2. Области видимости

Отдельные области видимости создают:

- тело функции;
- любой блок `{ ... }`;
- пространство имён `namespace`;
- объявление структуры — только для имён её полей как части описания типа.

### 3.3. Shadowing

Перекрытие имён (**shadowing**) разрешено только между разными вложенными областями видимости.

Пример допустимого кода:

```txt
let x = 1;
{
    let x = 2;
    print(x);
}
```

### 3.4. Запрет повторного объявления в одной области видимости

В одной и той же области видимости запрещено объявлять два объекта с одинаковым именем,
если они принадлежат к одной и той же категории имён и потенциально неоднозначны.

Запрещено:

- две локальные переменные с одним именем в одном блоке;
- параметр и локальная переменная с одинаковым именем в одной функции без вложенного блока;
- два поля структуры с одинаковым именем;
- две функции с одинаковым именем в одном пространстве имён (перегрузка отсутствует);
- две структуры с одинаковым именем в одной области;
- два синонима типов с одинаковым именем в одной области;
- два пространства имён с одинаковым именем в одной области.

### 3.5. Использование до объявления

Использование идентификатора до его объявления запрещено.

Это правило относится к:

- локальным переменным;
- функциям;
- структурам;
- type alias;
- пространствам имён.

Следовательно, семантика программы определяется с учётом текстового порядка объявлений.

---

## 4. Семантика типов и значений

### 4.1. Общий принцип

Каждое выражение имеет ровно один статический тип.

Если тип выражения невозможно определить однозначно, программа считается некорректной.

### 4.2. Категории значений

В языке различаются:

- **rvalue** — обычное вычисленное значение;
- **lvalue** — сущность, в которую можно присвоить значение.

`lvalue` образуют:

- имя мутабельной переменной `var`;
- доступ к полю мутабельной структуры;
- индексирование мутабельного массива.

`let` не образует присваиваемый `lvalue`.

### 4.3. Value semantics

Любое присваивание и передача аргумента функции копируют значение.

Примеры:

- присваивание массива копирует весь массив;
- присваивание структуры копирует все её поля;
- передача структуры в функцию не даёт возможности изменять исходный объект вызывающего кода.

### 4.4. Инициализация переменных

Использование неинициализированной переменной является ошибкой компиляции.

В языке **Astra** это правило обеспечивается синтаксически:

- каждая переменная объявляется только с обязательным инициализатором;
- отдельной инструкции объявления без значения не существует.

Следовательно, после успешного разбора и семантической проверки любая локальная переменная
всегда имеет начальное значение.

---

## 5. Семантика переменных

### 5.1. Неизменяемые переменные

`let` создаёт неизменяемую переменную.

Пример:

```txt
let x = 10;
```

После инициализации повторное присваивание запрещено.

### 5.2. Изменяемые переменные

`var` создаёт изменяемую переменную.

Пример:

```txt
var x: int32 = 10;
x = 20;
```

### 5.3. Время жизни

Переменная существует с момента завершения её объявления до выхода из блока,
в котором она объявлена.

Если выполнение покидает блок, все его локальные переменные уничтожаются.

---

## 6. Семантика функций

### 6.1. Объявление функции

Функция имеет:

- имя;
- список параметров;
- тип результата;
- тело.

Типы параметров и результата являются частью сигнатуры.

### 6.2. Вызов функции

При вызове функции:

1. аргументы вычисляются слева направо;
2. каждое вычисленное значение копируется в соответствующий параметр;
3. тело функции выполняется в новой области видимости.

Количество аргументов и их типы должны в точности соответствовать параметрам.

### 6.3. Возврат из функции

`return expr;` завершает выполнение функции и возвращает значение выражения.

`return;` разрешён только в функции с типом результата `unit`.

Если функция имеет тип результата, отличный от `unit`, то:

- каждый исполнимый путь должен завершаться `return expr;`, либо
- выполнение должно гарантированно завершаться через вызов `exit` или `panic`.

### 6.4. Функция `main`

Программа обязана содержать функцию:

```txt
fn main() -> int32
```

без параметров.

Возвращаемое значение `main` становится кодом завершения процесса.

---

## 7. Семантика структур

### 7.1. Именованные пользовательские типы

`struct` создаёт новый именованный составной тип.

```txt
struct Point {
    x: int32;
    y: int32;
}
```

Имя структуры участвует в номинальной типизации.

### 7.2. Поля структуры

Каждое поле имеет:

- уникальное имя внутри структуры;
- фиксированный тип;
- фиксированный порядок хранения.

### 7.3. Литерал структуры

Литерал структуры:

```txt
Point { x: 1, y: 2 }
```

корректен тогда и только тогда, когда:

1. указанный тип действительно является типом структуры;
2. перечислены все поля структуры;
3. отсутствуют лишние поля;
4. каждое поле упомянуто ровно один раз;
5. тип выражения каждого поля совпадает с объявленным типом этого поля.

Порядок полей в литерале может не совпадать с порядком объявления,
но семантически значения сопоставляются по именам.

### 7.4. Доступ к полю

Выражение `obj.field` корректно, если тип `obj` — структура, содержащая поле `field`.

Типом выражения является тип соответствующего поля.

Если `obj` — мутабельный `lvalue`, то `obj.field` также является мутабельным `lvalue`.

---

## 8. Семантика массивов

### 8.1. Тип массива

Массив имеет тип вида `[T; N]`, где:

- `T` — тип элемента;
- `N` — положительная целочисленная константа, часть типа.

### 8.2. Литерал массива

Литерал массива:

```txt
[1, 2, 3]
```

семантически корректен, если:

1. все элементы имеют один и тот же тип;
2. ожидаемый тип известен из контекста либо размер литерала определяется по числу элементов.

Если литерал используется без ожидаемого типа, его размер определяется длиной списка,
а тип элемента — по типу первого элемента с последующей проверкой остальных.

Пустой литерал массива без ожидаемого контекста запрещён.

### 8.3. Индексирование

Выражение `arr[i]` корректно, если:

- `arr` имеет тип массива `[T; N]`;
- `i` имеет целочисленный тип.

Тип выражения — `T`.

Если `arr` — мутабельный `lvalue`, тогда `arr[i]` тоже является мутабельным `lvalue`.

### 8.4. Ошибка времени выполнения: выход за границы

Если во время выполнения индекс выходит за диапазон `0 .. N-1`, программа обязана завершиться с сообщением:

```txt
runtime error: array index out of bounds at line <N>
```

Номер строки берётся из позиции операции индексирования в исходном коде.

---

## 9. Семантика type alias

Объявление

```txt
type Meters = int32;
```

создаёт **синоним типа**, но **не новый тип**.

Следствия:

- `Meters` и `int32` полностью совместимы;
- приведение между ними не требуется;
- в диагностике компилятор может отображать как исходное имя, так и развернутый тип.

---

## 10. Семантика пространств имён

`namespace` создаёт именованную область видимости для группировки объявлений.

Доступ к членам пространства имён осуществляется через `::`.

Пример:

```txt
Math::abs(10)
Geometry::Point { x: 1, y: 2 }
```

Разрешение `A::B` выполняется слева направо:

1. находится пространство имён или тип `A`;
2. внутри него разрешается имя `B`.

Для пространств имён доступны только верхнеуровневые объявления.

---

## 11. Семантика выражений

### 11.1. Литералы

- целый литерал — значение соответствующего целого типа;
- вещественный литерал — значение соответствующего вещественного типа;
- `true`, `false` — значения типа `bool`;
- строковый литерал — значение типа `string`.

### 11.2. Арифметические операции

Операции `+`, `-`, `*`, `/`, `%` определены только для числовых типов.

Дополнительно:

- `+` также определён для `string` как конкатенация;
- `%` определён только для целочисленных типов.

Операнды арифметической операции обязаны иметь **одинаковый тип**.
Неявные расширения и сужения отсутствуют.

### 11.3. Логические операции

- `!` принимает `bool` и возвращает `bool`;
- `&&` и `||` принимают `bool` и возвращают `bool`.

`&&` и `||` используют **короткое замыкание**:

- для `a && b` выражение `b` не вычисляется, если `a == false`;
- для `a || b` выражение `b` не вычисляется, если `a == true`.

### 11.4. Сравнения

Операции `==` и `!=` определены для:

- числовых типов;
- `bool`;
- `string`;
- одинаковых типов структур;
- одинаковых типов массивов.

Для структур и массивов сравнение выполняется покомпонентно.

Операции `<`, `<=`, `>`, `>=` определены только для числовых типов.

Результат любого сравнения имеет тип `bool`.

### 11.5. Унарный минус

Унарный `-` определён только для числовых типов и сохраняет тип операнда.

### 11.6. Явное приведение типов

Выражение

```txt
expr as T
```

выполняет явное приведение к типу `T`.

Допустимые приведения:

- между любыми целочисленными типами;
- между любыми вещественными типами;
- между целочисленными и вещественными типами;
- между type alias и их базовым типом.

Недопустимые приведения:

- к `string` и из `string`, кроме тождественного;
- между структурами разных типов;
- между массивами разных типов;
- между `bool` и числовыми типами.

### 11.7. Порядок вычисления

Порядок вычисления операндов и аргументов — строго **слева направо**,
за исключением короткого замыкания `&&` и `||`.

---

## 12. Семантика инструкций

### 12.1. Блок

Блок выполняет инструкции последовательно.

Если при выполнении инструкции происходит `return`, `break` или `continue`,
оставшаяся часть блока не выполняется.

### 12.2. Пустая инструкция

Инструкция `;` не производит никакого эффекта.

### 12.3. Условный оператор

`if (cond) then_block else else_block`

Сначала вычисляется `cond`. Оно должно иметь тип `bool`.

- если `cond == true`, выполняется `then_block`;
- иначе, если присутствует `else`, выполняется `else_block`.

### 12.4. Цикл while

`while (cond) body`

Семантика:

1. вычислить `cond`;
2. если `cond == false`, завершить цикл;
3. выполнить `body`;
4. повторить.

Условие должно иметь тип `bool`.

### 12.5. Break

`break;` немедленно завершает ближайший охватывающий цикл `while`.

Использование `break` вне цикла является ошибкой компиляции.

### 12.6. Continue

`continue;` прекращает текущую итерацию ближайшего охватывающего цикла
и переходит к повторной проверке условия.

Использование `continue` вне цикла является ошибкой компиляции.

### 12.7. Присваивание

Инструкция присваивания корректна, если:

1. левая часть является `lvalue`;
2. левая часть мутабельна;
3. тип правой части в точности совпадает с типом левой части.

Неявные приведения запрещены.

### 12.8. Инструкция-выражение

Как инструкция разрешаются только выражения, у которых допустимы побочные эффекты.

Практически это означает, что компилятор обязан принимать как минимум:

- вызовы пользовательских функций;
- вызовы встроенных функций.

Использование чисто вычислительного выражения как инструкции допустимо,
но компилятор может выдавать предупреждение, если механизм предупреждений реализован.

---

## 13. Встроенные функции и побочные эффекты

### 13.1. `print`

```txt
print(value) -> unit
```

Печатает значение в стандартный поток вывода.

Допустимые аргументы:

- любой встроенный числовой тип;
- `bool`;
- `string`.

Печать структур и массивов в базовой версии не обязана поддерживаться.
Если реализация не поддерживает такие типы, вызов считается ошибкой компиляции.

### 13.2. `input`

```txt
input() -> string
```

Считывает одну строку из стандартного потока ввода.

### 13.3. `exit`

```txt
exit(code: int32) -> unit
```

Немедленно завершает программу с указанным кодом возврата.
Последующий код недостижим.

### 13.4. `panic`

```txt
panic(message: string) -> unit
```

Немедленно аварийно завершает программу,
выводя сообщение об ошибке времени выполнения.
Последующий код недостижим.

---

## 14. Ошибки времени выполнения

В базовой версии языка компилятор обязан обеспечить обработку двух ошибок времени выполнения.

### 14.1. Деление на ноль

При выполнении операций `/` и `%`, если правый операнд равен нулю,
программа обязана завершиться сообщением:

```txt
runtime error: division by zero at line <N>
```

### 14.2. Выход за границы массива

При обращении `arr[i]`, если индекс лежит вне допустимых границ,
программа обязана завершиться сообщением:

```txt
runtime error: array index out of bounds at line <N>
```

### 14.3. Источник номера строки

Во всех runtime-ошибках строка `<N>` берётся из позиции соответствующего AST-узла:

- для деления на ноль — из оператора `/` или `%`;
- для выхода за границы — из операции индексирования.

---

## 15. Семантические ошибки компиляции

Семантический анализатор обязан диагностировать как минимум следующие ошибки:

1. использование необъявленного идентификатора;
2. использование идентификатора до объявления;
3. повторное объявление в одной области видимости;
4. отсутствие `main` или неверная сигнатура `main`;
5. несовместимость типов в присваивании;
6. несовместимость типов в `return`;
7. неверное число аргументов функции;
8. несовместимые типы аргументов функции;
9. обращение к несуществующему полю структуры;
10. инициализация структуры неполным или лишним набором полей;
11. использование `break` или `continue` вне цикла;
12. использование `return;` в функции с невозвращаемым типом, отличным от `unit`;
13. отсутствие обязательного возврата в функции с не-`unit` результатом;
14. присваивание в `let`;
15. попытка присваивания в немутируемый результат;
16. недопустимое приведение типов;
17. индексирование не-массива;
18. вызов не-функции;
19. использование типа как значения и значения как типа;
20. объявление массива с неположительным размером.

---

## 16. Дополнительная возможность: вывод типов для переменных

### 16.1. Общее правило

Если переменная объявлена как

```txt
let x = expr;
```

то тип `x` выводится из статического типа `expr`.

### 16.2. Ограничения

Вывод типов разрешён только для `let`.

Для `var` тип должен быть указан явно. Это упрощает реализацию мутабельности
и делает семантику присваивания прозрачной.

### 16.3. Примеры

```txt
let a = 10;          // a: int32
let b = true;        // b: bool
let c = "hello";    // c: string
let d = [1, 2, 3];   // d: [int32; 3]
```

Если тип инициализатора не может быть выведен однозначно, программа некорректна.

## 17. Достижимость и корректность управления потоком

Семантический анализатор обязан проверять базовую корректность управления потоком.

### 17.1. Недостижимый код

После инструкций

- `return`
- `break`
- `continue`
- `exit(...)`
- `panic(...)`

последующие инструкции в том же блоке считаются недостижимыми.

Обнаружение недостижимого кода может быть:

- либо ошибкой компиляции,
- либо предупреждением.

Для базовой версии достаточно хотя бы корректно обрабатывать такие конструкции при анализе потока.

### 17.2. Проверка возврата значения

Для функции с результатом, отличным от `unit`, анализатор обязан подтвердить,
что все достижимые пути исполнения завершаются возвратом значения или безусловным завершением программы.

## 18. Семантический пример

```txt
struct Pair {
    left: int32;
    right: int32;
}

fn sum_pair(p: Pair) -> int32 {
    return p.left + p.right;
}

fn main() -> int32 {
    let p = Pair { left: 10, right: 20 };
    let xs = [1, 2, 3];
    var i: int32 = 0;
    var acc: int32 = 0;

    while (i < 3) {
        acc = acc + xs[i];
        i = i + 1;
    }

    print(sum_pair(p));
    print(acc);
    return 0;
}
```

Семантически здесь происходит следующее:

1. объявляется новый номинальный тип `Pair`;
2. функция `sum_pair` принимает копию `Pair` и возвращает `int32`;
3. `p` получает тип `Pair` через вывод типа;
4. `xs` получает тип `[int32; 3]` через вывод типа;
5. цикл суммирует элементы массива с проверкой границ во время выполнения;
6. вызовы `print` порождают побочные эффекты;
7. `main` возвращает код завершения `0`.
