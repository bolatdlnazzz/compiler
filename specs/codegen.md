# codegen.md — генерация кода для языка **Astra**

## 1. Назначение документа

Документ описывает backend компилятора языка **Astra**:

- выбранную целевую платформу;
- соглашение о вызовах;
- представление значений в памяти;
- правила генерации кода для конструкций языка;
- промежуточное представление (IR);
- реализуемую оптимизацию.

Документ написан так, чтобы по нему можно было реализовать кодогенератор независимо от авторского кода.

---

## 2. Выбор платформы

Для языка **Astra** выбирается следующий вариант backend-а:

> **Нативная генерация x86-64 ассемблера** под **System V AMD64 ABI**.

Это решение соответствует базовым требованиям ТЗ, где разрешена нативная генерация x86-64 asm.

### 2.1. Обоснование выбора

1. backend остаётся полностью под контролем проекта;
2. не требуется зависимость от LLVM;
3. легче показать связь между AST/IR и конечным машинным кодом;
4. удобно реализовать базовый набор конструкций учебного языка.

### 2.2. Формат выходного файла

Генератор создаёт текстовый файл с x86-64 assembly в синтаксисе NASM.

Далее сборка исполняемого файла производится внешним инструментом, например:

```txt
nasm -felf64 out.asm -o out.o
clang out.o runtime.o -o out
```

или эквивалентной командой для целевой ОС.

---

## 3. Общая архитектура backend-а

Полный pipeline компилятора:

```txt
source
  -> Lexer
  -> Parser
  -> Semantic Analyzer
  -> Lowering to IR
  -> IR Optimizer
  -> x86-64 Code Generator
  -> executable
```

Наличие IR является **дополнительной возможностью** сверх обязательной базы.

### 3.1. Почему вводится IR

IR нужен, чтобы:

- отделить семантику языка от деталей машинной архитектуры;
- упростить генерацию переходов и временных значений;
- сделать возможной локальную оптимизацию (constant folding);
- печатать промежуточное представление для отладки.

---

## 4. Модель времени выполнения

### 4.1. Общая память программы

Программа использует:

- **стек вызовов** — параметры, локальные переменные, временные значения фиксированного размера;
- **статический сегмент данных** — строковые литералы и константные данные;
- **runtime-библиотеку** — для ввода/вывода, ошибок времени выполнения и работы со строками.

### 4.2. Динамическая память

В базовой версии языка динамические массивы отсутствуют.

Строки реализуются через runtime и хранятся как ссылочное runtime-значение,
но на уровне языка остаются обычным значением со строгой типизацией.

---

## 5. Представление значений

### 5.1. Скалярные типы

| Тип | Машинное представление |
|-----|-------------------------|
| `int8`, `uint8`, `bool` | 1 байт |
| `int16`, `uint16` | 2 байта |
| `int32`, `uint32` | 4 байта |
| `int64`, `uint64` | 8 байт |
| `float32` | 4 байта |
| `float64` | 8 байт |
| `unit` | не хранится |
| `string` | 8-байтный дескриптор/указатель на runtime-объект |

### 5.2. Массивы

Массив `[T; N]` хранится **inline**, как непрерывный блок из `N` элементов типа `T`.

Размер массива:

```txt
size([T; N]) = N * size(T)
```

### 5.3. Структуры

Структура хранится **inline** как последовательность полей в порядке объявления.

#### Выравнивание

Для простоты реализации применяется естественное выравнивание каждого поля по его размеру,
но не более 8 байт.

Размер структуры округляется вверх до максимального выравнивания её полей.

### 5.4. Alias

`type alias` не влияет на layout и отсутствует в runtime-представлении.

---

## 6. Runtime-поддержка

Кодогенератор опирается на небольшой runtime, реализованный на C/C++ или asm.

### 6.1. Обязательные runtime-функции

```txt
astra_print_i32
astra_print_i64
astra_print_u32
astra_print_u64
astra_print_f32
astra_print_f64
astra_print_bool
astra_print_string
astra_input_string
astra_exit
astra_panic
astra_string_concat
astra_string_eq
astra_string_ne
astra_string_len
astra_rt_div_zero
astra_rt_oob
```

Допускается иная система имён, если семантика эквивалентна.

### 6.2. Назначение runtime-функций

- печать скалярных значений;
- ввод строки;
- аварийное завершение программы;
- поддержка строковых операций;
- единая обработка runtime-ошибок.

---

## 7. Соглашение о вызовах

Используется **System V AMD64 ABI**.

### 7.1. Передача целочисленных и указательных аргументов

Первые 6 аргументов целочисленного или адресного класса передаются через регистры:

```txt
RDI, RSI, RDX, RCX, R8, R9
```

### 7.2. Передача аргументов с плавающей точкой

Первые 8 вещественных аргументов передаются через:

```txt
XMM0..XMM7
```

### 7.3. Возврат значений

- целочисленные и указательные значения до 64 бит возвращаются в `RAX`;
- `float32`/`float64` — в `XMM0`;
- агрегаты большого размера возвращаются через скрытый указатель на буфер результата.

### 7.4. Возврат структур и массивов

Для структур и массивов, не помещающихся в регистр,
используется скрытый первый аргумент — адрес памяти результата.

Фактически функция:

```txt
fn make_point() -> Point
```

на ABI-уровне реализуется как:

```txt
void make_point(Point* out)
```

---

## 8. Организация стекового кадра

### 8.1. Общая схема

Каждая функция создаёт стековый кадр следующего вида:

```txt
| caller data          |
| return address       |
| old RBP              |
| spilled registers    |
| local variables      |
| temporaries          |
| alignment padding    |
```

Пролог функции:

```asm
push rbp
mov rbp, rsp
sub rsp, <frame_size>
```

Эпилог:

```asm
mov rsp, rbp
pop rbp
ret
```

### 8.2. Локальные переменные

Каждая локальная переменная получает фиксированное смещение от `RBP`.

Пример:

```txt
x  -> [rbp - 8]
y  -> [rbp - 16]
```

### 8.3. Временные значения

Временные значения после lowering в IR обычно хранятся виртуально,
а на этапе конкретного codegen либо:

- размещаются в регистрах,
- либо проливаются в стек.

Для семестрового проекта допустим простой allocator:

- выражение вычисляется в один-два временных регистра;
- при нехватке значений используется стек.

---

## 9. Промежуточное представление (IR)

## 9.1. Общие свойства IR

IR языка **Astra**:

- линейный;
- трёхадресный;
- типизированный;
- разбит на basic blocks;
- пригоден для печати в читаемом виде.

### 9.2. Структура IR-функции

IR-функция состоит из:

- сигнатуры;
- списка локальных temporaries;
- набора basic blocks;
- терминатора в конце каждого блока.

Пример псевдо-IR:

```txt
fn sum(a: int32, b: int32) -> int32
entry:
    %0 = load_arg 0
    %1 = load_arg 1
    %2 = add_i32 %0, %1
    ret %2
```

### 9.3. Базовые IR-инструкции

#### Значения и перемещения

```txt
const_i32 k
const_i64 k
const_f64 k
copy dst, src
```

#### Арифметика

```txt
add
sub
mul
div
mod
neg
```

#### Логика и сравнения

```txt
and
or
not
cmp_eq
cmp_ne
cmp_lt
cmp_le
cmp_gt
cmp_ge
```

#### Память и адресация

```txt
alloc_local
load
store
field_ptr
index_ptr
```

#### Управление

```txt
jump label
branch cond, then_label, else_label
call
ret
```

#### Приведения

```txt
cast
```

### 9.4. Lowering AST -> IR

#### Литералы

Литералы переводятся в `const_*`.

#### Бинарные выражения

Каждое бинарное выражение понижает оба операнда в temporaries,
после чего генерируется одна инструкция для результата.

#### `if`

`if` переводится в три блока:

```txt
cond_block
then_block
else_block / join_block
```

#### `while`

`while` переводится в четыре блока:

```txt
preheader
cond
body
exit
```

`break` переходит в `exit`, `continue` — в `cond`.

#### Функции

Аргументы функции становятся IR-значениями или адресами, доступными в entry-block.

## 10. Оптимизация IR

В проекте реализуется одна обязательная дополнительная оптимизация:

> **constant folding**

### 10.1. Что делает constant folding

Если выражение целиком вычислимо на этапе компиляции,
оно заменяется константой.

Примеры:

```txt
2 + 3 * 4   -> 14
10 > 5      -> true
!false      -> true
```

### 10.2. Где применяется

Оптимизация выполняется после lowering и до machine code generation.

### 10.3. Ограничения

Constant folding применяется только к:

- арифметике над константами;
- логике над константами;
- сравнениям констант;
- явным приведениям констант.

Операции с побочными эффектами не сворачиваются.

---

## 11. Генерация кода по типам узлов

### 11.1. Числовые литералы

Числовой литерал генерируется как загрузка константы в регистр
или как непосредственный операнд инструкции.

Пример:

```txt
let x = 5;
```

может перейти в:

```asm
mov dword [rbp-4], 5
```

### 11.2. Локальная переменная

Для локальной переменной резервируется место в стеке.
Инициализатор вычисляется и сохраняется по смещению этой переменной.

### 11.3. Присваивание

1. вычислить адрес lvalue;
2. вычислить rhs;
3. записать rhs по адресу lvalue.

Для агрегатов (структур и массивов) выполняется поэлементное или поблочное копирование.

### 11.4. Арифметические выражения

Для целых типов используются обычные инструкции:

```asm
add
sub
imul
idiv / div
```

Перед `/` и `%` кодогенератор обязан вставить проверку делителя на ноль.

### 11.5. Логические выражения

#### Обычные логические операции

Булевы значения хранятся как `0` или `1`.

#### Короткое замыкание

`&&` и `||` генерируются через ветвления, а не через простую арифметику над булевыми.

### 11.6. Сравнения

Сравнения генерируются через `cmp` + `setcc` либо эквивалентную схему.

Результат нормализуется к `0`/`1`.

### 11.7. Доступ к полю структуры

Адрес поля вычисляется как:

```txt
base_address + field_offset
```

Смещение берётся из layout структуры, определённого на этапе семантики/подготовки типов.

### 11.8. Индексирование массива

Адрес элемента вычисляется как:

```txt
base + index * element_size
```

Перед вычислением доступа обязателен runtime-check:

```txt
0 <= index < N
```

При нарушении вызывается `astra_rt_oob(line)`.

### 11.9. Вызов функции

1. вычислить аргументы слева направо;
2. разложить их по ABI-регистрам и стеку;
3. выполнить `call`;
4. извлечь результат из регистра(ов) или буфера результата.

### 11.10. Return

- скалярный результат помещается в регистр возврата;
- агрегатный результат записывается в скрытый буфер результата;
- выполняется переход в единый эпилог функции.

---

## 12. Генерация кода для строк

### 12.1. Представление строки

На уровне backend строка представляется 64-битным дескриптором,
с которым работает runtime.

### 12.2. Операции

- конкатенация `a + b` переводится в вызов `astra_string_concat`;
- `a == b` переводится в `astra_string_eq`;
- `a != b` переводится в `astra_string_ne`;
- `len(a)` переводится в `astra_string_len`;
- `print(a)` переводится в `astra_print_string`.

Строковые литералы размещаются в `.rodata`.

---

## 13. Генерация runtime-ошибок

### 13.1. Деление на ноль

Перед `div` / `mod` вставляется проверка делителя.

Пример схемы:

```asm
cmp rbx, 0
je .panic_div_zero
```

где `.panic_div_zero` вызывает runtime-функцию:

```txt
astra_rt_div_zero(line_number)
```

### 13.2. Выход за границы массива

Перед обращением `arr[i]` генерируется проверка диапазона.

Если индекс некорректен, вызывается:

```txt
astra_rt_oob(line_number)
```

### 13.3. Формат сообщений

Runtime обязан печатать сообщения в формате:

```txt
runtime error: division by zero at line <N>
runtime error: array index out of bounds at line <N>
```

---

## 14. Генерация кода для встроенных функций

### 14.1. `print`

В зависимости от статического типа аргумента выбирается соответствующая runtime-функция печати.

Примеры:

- `int32` -> `astra_print_i32`
- `bool` -> `astra_print_bool`
- `string` -> `astra_print_string`

### 14.2. `input`

Вызов `input()` переводится в вызов `astra_input_string`.

### 14.3. `exit`

`exit(code)` переводится в runtime-вызов `astra_exit(code)` и считается не возвращающим управление.

### 14.4. `panic`

`panic(message)` переводится в вызов `astra_panic(message)` и считается не возвращающим управление.

### 14.5. `len`

`len(s)` переводится в `astra_string_len(s)`.

---

## 15. Схема распределения регистров

Для упрощённой учебной реализации используется следующая стратегия.

### 15.1. Общий подход

- выражения вычисляются рекурсивно;
- для скалярных временных значений используются временно свободные регистры;
- при нехватке регистров значения сохраняются в стек.

### 15.2. Предпочтительные регистры

Для целых и адресных промежуточных значений:

```txt
RAX, RBX, RCX, RDX, R10, R11
```

Для вещественных:

```txt
XMM0..XMM5
```

### 15.3. Сохраняемые и разрушаемые регистры

Согласно ABI:

- caller-saved: `RAX`, `RCX`, `RDX`, `RSI`, `RDI`, `R8-R11`
- callee-saved: `RBX`, `RBP`, `R12-R15`

Если кодогенератор использует callee-saved регистр внутри функции,
он обязан сохранить и восстановить его в прологе/эпилоге.

## 16. Пример lowering и кодогенерации

### 16.1. Исходный код

```txt
fn add(a: int32, b: int32) -> int32 {
    return a + b;
}
```

### 16.2. IR

```txt
fn add(a: int32, b: int32) -> int32
entry:
    %0 = load_arg 0
    %1 = load_arg 1
    %2 = add_i32 %0, %1
    ret %2
```

### 16.3. Возможный NASM-вывод

```asm
global add
add:
    push rbp
    mov rbp, rsp
    mov eax, edi
    add eax, esi
    pop rbp
    ret
```

---

## 17. Генерация кода для управляющих конструкций

### 17.1. If / Else

Для `if` генерируются минимум три метки:

```txt
.then
.else
.end
```

Условие вычисляется в булево значение,
после чего `cmp`/`je` или аналогичная инструкция выбирает ветку.

### 17.2. While

Для `while` генерируются метки:

```txt
.cond
.body
.end
```

`break` прыгает в `.end`,
`continue` прыгает в `.cond`.

---

## 18. Ограничения реализации

Для семестровой реализации допускаются следующие ограничения:

1. печать массивов и структур не поддерживается;
2. строки реализуются через runtime, а не полностью вручную в asm;
3. оптимизации ограничиваются constant folding;
4. аллокация регистров может быть наивной;
5. все локальные агрегаты могут храниться в стеке без сложного register passing.

Эти ограничения не нарушают базовые требования ТЗ, если описаны в отчёте и спецификации.

---

## 19. Минимальные требования к корректности codegen

Кодогенератор обязан корректно поддерживать все обязательные конструкции базового языка:

- функции и `main`;
- параметры и возврат значений;
- локальные переменные `let` и `var`;
- арифметические, логические и сравнительные выражения;
- явный cast;
- массивы фиксированного размера и индексирование;
- структуры и доступ к полям;
- `if/else`, `while`, `break`, `continue`, `return`;
- вызовы встроенных функций;
- runtime-check для деления на ноль и выхода за границы массива.

## 20. Итог

Выбранный backend для языка **Astra** — это реалистичная и защищаемая архитектура для оценки 4–5:

- базовые требования ТЗ выполнены полностью;
- добавлена отдельная фаза lowering в IR;
- добавлена оптимизация constant folding;
- схема соглашения о вызовах, layout памяти и обработка ошибок описаны явно;
- по этой спецификации можно реализовать рабочий учебный компилятор на C++23.
